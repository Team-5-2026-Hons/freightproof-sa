**1\. Document & Manifest Flow**

A key clarification was provided on how documentation flows between origin and destination, and what the driver actually receives\.

**1\.1 The Linehaul Document \(Driver’s Document\)**

# The driver does NOT receive the cargo manifest\. This is a deliberate risk\-management decision: knowledge of high\-value contents \(e\.g\. cell phones, electronics\) significantly increases theft and security risk\.

# Instead, the driver is issued a Linehaul document by the truck owner \(LFG or equivalent\)\. This document contains:

## Type and configuration of the vehicle

## Seal numbers

## Vehicle registration number

## Driver details

## A consolidated unit count \(e\.g\. “1 x 8\-tonne load”\) but not the specific contents

# This linehaul is the driver’s core document and is currently printed manually — a key target for digitalisation in FreightProof\.

**1\.2 The Full Manifest**

# The full manifest \(e\.g\. listing all 100 parcels within the truck\) is communicated directly between operational/customer service teams at origin and destination \(e\.g\. FedEx Johannesburg to FedEx Durban\)\.

# It is generated by the customer’s own system \(e\.g\. Parcel Perfect for FedEx\) and does not pass through the driver\.

# LFG’s freight\-level document is their internal check\-and\-balance for what was loaded into their vehicle at origin\.

**2\. FreightProof System Design Implications**

**2\.1 Correct Architecture Confirmed**

# FreightProof is correctly positioned as the interface between origin and destination operations teams \(i\.e\. FedEx Joburg ↔ FedEx Durban\), not as a tool exposed to the driver at a parcel level\.

# The system should verify that X units/parcels were loaded into the truck \(chain\-of\-custody checkpoint\), then generate a consolidated vehicle document for the driver — a digital replacement for the current printed linehaul/waybill\.

**2\.2 Dispatcher Portal \(Control Hub\)**

# The dispatcher/control hub should have access to the full manifest and waybill details\.

# The dispatcher can view all client consignments, loading order, weights, and any exceptions\.

# Ciaran demonstrated the current dispatcher portal\. Bruce confirmed the timeline\-based layout is on the right track, comparing it favourably to an air freight incident report he uses internally\.

# Bruce highlighted that FreightProof’s ability to drill into any event on the timeline \(e\.g\. loading, transit exceptions, delivery\) is a differentiator over existing market systems, which cannot perform event\-level manifest investigation\.

**2\.3 Driver App**

# Tim is building the driver app in Iteration 2\.

# The driver app must show only limited, consolidated information — a digital version of the linehaul document:

## Vehicle type and configuration

## Seal numbers

## Registration number

## Driver details

## Unit count \(total consolidated units\) — not the itemised cargo contents

# Showing cargo contents to the driver is explicitly not desired and poses a security risk\.

**3\. Load Configuration & Weight Management**

Bruce provided a detailed explanation of load configuration requirements, which should inform how the system validates and records loading events\.

**3\.1 Load Trimming**

# It is critical that weight is distributed correctly across the vehicle: more scale/actual weight should sit over the rear axle, not between the front and rear axles\.

# This is referred to as ‘trimming the load’ and is enforced via a vehicle blueprint/load configuration spec that LFG provides to clients\.

**3\.2 Maximum Allowable Weight**

# The system should ensure that the consolidated load does not exceed the vehicle’s maximum legal weight\.

# South African roads have weigh bridges that enforce limits\. Non\-compliance can result in:

## Fines for the operator

## Confiscation of the vehicle at the weigh bridge

## Emergency recovery involving forklifts and support vehicles to offload and reload on the side of the road

# All fines and recovery costs arising from incorrect loading are passed back to the loading client \(e\.g\. FedEx\), as they are responsible for adhering to LFG’s load configuration blueprint\.

**3\.3 Responsibility**

# Loading is FedEx’s responsibility\. LFG provides them with the vehicle blueprint \(including high\-canopy configurations for volumetric freight\), and FedEx must load accordingly\.

# It is in the client’s financial interest to comply with the load configuration, as non\-compliance costs fall on them\.

**4\. Multiple Clients & Break Bulk Operations**

**4\.1 Design Requirement: Multiple Clients per Trip**

# Confirmed: the system must be designed to accommodate multiple clients on a single truck\. This is standard practice for break bulk / break box services\.

# Example: a truck departing Durban may carry loads from FedEx, Courier Guy, Seaborne, and others simultaneously\.

# LFG Durban manifests all clients onto one vehicle internally; this manifest is communicated LFG Durban → LFG Johannesburg \(not to the driver\)\.

**4\.2 Loading Order & Priority**

# Loading order is determined by urgency, yield, and client priority\. For example, a more urgent FedEx load is placed at the door \(unloaded first on arrival\), while a lower\-priority Courier Guy load is placed at the bulkhead \(driver\-side, rear of load body\)\.

# This operational priority logic should be accessible to the dispatcher in FreightProof\.

**4\.3 Two Service Models**

# Scheduled Break Bulk Service: Clients bring their shipments to the LFG facility\. LFG consolidates and loads the truck according to priority and weight rules\.

# Ad Hoc Collection Service: The truck travels to individual client sites to collect loads \(e\.g\. collect 3 tonnes from FedEx, then continue to Courier Guy for 4 tonnes\)\. This is a secondary, on\-request service\.

# The system must therefore support multiple pickup locations and multiple drop\-off points per trip\.

**5\. Return Legs**

# Return legs must be tracked distinctly from outbound trips\. If the vehicle returns empty, that cost is absorbed by LFG, making it important for management to have visibility\.

# LFG actively tries to find return loads to minimise empty\-leg costs\.

# Tracking return legs also feeds into rotation planning \(i\.e\. ensuring a truck is available and positioned for the following night’s departing service\)\.

# Design decision: the dispatcher should be able to either initiate a return leg as a continuation of an existing trip or create a new trip entry\. This still needs to be finalised in the system design\.

**6\. Parcel\-Level & Unit\-Level Tracking**

**6\.1 Importance**

# Parcel\-level tracking is very important\. A key use case is dispute resolution: if a client \(e\.g\. FedEx\) reports a missing parcel, LFG needs to determine whether the parcel was in their custody and, if so, at which point in the journey it left\.

# FreightProof’s ability to search a specific parcel reference and trace it through loading events, in\-transit checkpoints, and delivery handshakes was confirmed as a significant capability gap in the current market\.

**6\.2 Consolidation Constraint**

# In practice, FedEx consolidates multiple parcels onto wooden pallets \(e\.g\. 50 parcels per pallet, shrink\-wrapped\)\. LFG receives these as single units\.

# LFG therefore cannot track at individual parcel level — only at the consolidated unit/pallet level\.

# FreightProof should track at the unit level \(i\.e\. the pallet or consolidated shipment handed to LFG\), not at the individual parcel level\. FedEx’s internal systems \(Parcel Perfect\) hold the parcel\-to\-pallet mapping\.

# Example: a 10\-pallet shipment may contain parcels from multiple FedEx clients across different pallets\. LFG tracks each pallet as a unit\.

**7\. Dispatcher Portal Demo & Bruce’s Feedback**

# Ciaran demonstrated the current dispatcher portal \(Iteration 1 build\)\.

# Bruce confirmed the timeline\-based trip view is consistent with industry documentation formats he uses in air freight incident reporting\.

# Key UI feedback: the ability to click into each timeline event \(e\.g\. truck loaded, in\-transit, exception\) and drill into the underlying manifest data, consignment numbers, and coordinates is a strong differentiator\.

# Bruce suggested a future enhancement: integrating the vehicle’s onboard camera system so that when an exception is flagged, the dispatcher \(and client\) can pull a live or recorded camera feed directly within the exception event view\.

# Ciaran raised the possibility of automatically saving a short video clip \(e\.g\. 5 minutes before and after an exception event\) from the Pulsit camera system\. Bruce agreed this would be a valuable feature and committed to raising it with Pulsit as a future enhancement request\.

# The exceptions feature \(currently logging out\-of\-route events with GPS coordinates\) was well\-received\.

**8\. Development Iteration Plan**

# Iteration 1 \(Complete\): Dispatcher portal — trip creation, blockchain recording of events, timeline view, handshake events, exception logging\.

# Iteration 2 \(In Progress\): Dispatcher portal \+ driver app integration\. The goal is to mock the full end\-to\-end flow from trip creation through to proof of delivery, without live API connections\.

# The team is not attempting real API integrations in Iteration 2\. The focus is on visualising the complete process and validating the data model\.

# An important output of Iteration 2 is identifying the exact data fields needed from Parcel Perfect, so the team knows precisely what to request when the API negotiation begins\.

**9\. Parcel Perfect API Integration**

# Parcel Perfect is used by FedEx \(and other clients\) to generate manifests\. It is a widely\-used SA freight management system, but the developer is protective of their API and documentation is limited publicly\.

# Bruce confirmed that integration is achievable but requires a three\-party discussion: FreightProof \(UCT team\) \+ FedEx \(Parcel Perfect’s paying customer\) \+ Parcel Perfect \(API owner\)\.

# FedEx would instruct Parcel Perfect to share the API, and Parcel Perfect would then review FreightProof’s requirements before granting access\.

# Key framing for the negotiation: FreightProof only reads from Parcel Perfect — it does not write, does not compete with Parcel Perfect’s business, and effectively adds value to the Parcel Perfect ecosystem\. This reduces the perceived threat and strengthens the case\.

# The same dynamic applies to other TMS/WMS providers \(e\.g\. ShipMate\) that may be integrated in the future\.

# This will need to be tackled in a later iteration once the mock\-up is sufficiently mature to demonstrate value to all parties\.
