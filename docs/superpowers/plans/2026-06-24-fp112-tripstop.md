## A. Design

### A.1 The key realisation

Multi-client and multi-origin/destination are **already half-modelled**: `Consignment`
(`trips.py:46`) carries its own `client_organization_id`, `origin_precinct_id`,
`destination_precinct_id`, `slot_time_origin`, `slot_time_destination`, and a nullable
`trip_id`. What's missing is:

1. `Trip` hard-codes a **single** client + single origin + single destination (all `nullable=False`,
   `trips.py:108–122`), which contradicts a multi-client/multi-stop trip.
2. There is **no sequenced stop / leg** concept — no ordering, no per-stop slot, and no way to pin a
   consignment's origin/destination to a position in the route.
3. The journey-lock hash (`crypto/hashing.py`) covers only origin + destination precincts, so a
   multi-stop route would not be tamper-evident. **(→ FP-113, not this ticket.)**

### A.2 Target shape

A **stop has no inherent origin/destination type** — the *same* physical stop can be an origin for
one consignment and a destination for another (see `docs/glossary.md` §3). So the origin/destination
**role lives on the consignment→stop link**, not on the stop. A `TripStop` is just a sequenced
*visit to a precinct*.

```
Trip (1) ──< TripStop (N, ordered by sequence)       # the route the truck drives (multiple O/D)
Trip (1) ──< Consignment (N, one client each)        # the bookings; already exists
Consignment ──> pickup_stop   (its ONE origin)       # FK → TripStop
Consignment ──> delivery_stop (its ONE destination)  # FK → TripStop
```

**New model `TripStop`** (`trip_stops`) — a pure route waypoint, **no type enum needed**:

| Field | Type | Notes |
|---|---|---|
| `id` | UUID PK | |
| `trip_id` | FK → `trips.id`, not null | |
| `precinct_id` | FK → `precincts.id`, not null | the physical location visited |
| `sequence` | int, not null | route order, 0-based; unique per `(trip_id, sequence)` |
| `slot_time` | datetime, nullable | per-stop SLA slot |
| `notes` | str(255), nullable | |
| `created_at` / `updated_at` | datetime | per CLAUDE.md every table has both |

A stop's **role is derived**, not stored: origin for every consignment whose `pickup_stop_id` points
at it, destination for every consignment whose `delivery_stop_id` points at it — possibly **both**.

**`Consignment` additions** (keep existing `origin/destination_precinct_id` — they remain the
intrinsic booking location, valid even before a trip is assigned):
- `pickup_stop_id` → FK `trip_stops.id`, nullable — the stop realising its origin on this trip
- `delivery_stop_id` → FK `trip_stops.id`, nullable — the stop realising its destination
- `load_priority` → int, nullable — **recorded evidence only** (door vs bulkhead, Bruce 24 Jun);
  lives here because it's *this client's freight position*. FreightProof records, does not enforce
  (`scope-boundaries.md` §3).

**Validation rule** (assigned consignments): `pickup_stop.sequence < delivery_stop.sequence`, and
each linked stop's `precinct_id` should match the consignment's corresponding precinct.

**`Trip` relaxations** (do **not** drop — relax to nullable for back-compat):
- `client_organization_id` → nullable (client now lives per-consignment; null = multi-client trip)
- `origin_precinct_id`, `destination_precinct_id` → nullable, treated as **derived convenience**
  (= precinct of the earliest / latest stop) so existing reads keep working.

### A.3 Backward compatibility (demo stays single-client/single-leg)

The iter-2 demo is single-client JHB→DBN. To avoid breaking the existing create path and the
dispatcher (FP-114 lands later):

- `TripCreateRequest` gains an **optional** `stops: list[TripStopCreate]`.
- If `stops` is **omitted** but `origin_precinct_id` + `destination_precinct_id` are given,
  `create_trip` **synthesises** two stops (`seq 0` @ origin precinct, `seq 1` @ destination
  precinct), links any single consignment's `pickup_stop`→seq 0 / `delivery_stop`→seq 1, and
  back-fills `Trip.origin/destination_precinct_id`. → old single-leg callers are unchanged.
- If `stops` is **provided**, persist them in order and derive `Trip.origin/destination` from the
  earliest / latest stop.

### A.5 Out of scope (this ticket)

- Journey-lock-hash payload changes → **FP-113**.
- Dispatcher multi-stop wizard UI → **FP-114**.
- **Per-stop handshakes.** The current 5-handshake lifecycle (`HandshakeType`, `enums.py:33`) is
  **single-origin/single-destination shaped** (one gate-in / load / unload). True mid-route
  load/unload makes each stop a seal break + reseal needing its **own handshake** — a structural
  change beyond this ticket. FP-112 lands the *data shape* only; the iter-2 demo runs the single-seal
  depot-to-depot path. **Dependency/follow-on: a new per-stop-handshake ticket (recommend creating
  it), bundled with or after FP-113.** Design ideas captured in
  `docs/design-notes/2026-06-24-multi-stop-handshakes.md`; see also `docs/glossary.md` §4.
- Sub-consignment `HandlingUnit`/pallet vs unit-count leaf → blocked on July site visit
  (`parcel-traceability.md` §8).

### A.6 Trip "types" — one model, two wizard flows (informs FP-114)

A single-origin/single-destination trip is the **degenerate case** of a multi-stop trip (2 stops,
1 consignment) — exactly what the back-compat synthesis in A.3 produces. **Do not add a `trip_type`
field or a second model.** The dispatcher wizard (FP-114) may offer a *simple* flow (one origin, one
destination — the iter-2 path) and an *advanced* flow (add stops/consignments), but **both write the
same `Trip` / `TripStop` / `Consignment` rows**. "Direct vs multi-drop" is a *derived label* (stop
count), never stored. This keeps validation, hashing, and handshakes on a single code path and lets a
direct trip gain a stop later without a type migration.

### Steps

1. **Model — `TripStop`.** Add the class to `trips.py` per A.2 (FKs to `trips` + `precincts`,
   `sequence`, nullable `slot_time`/`notes`, `created_at`/`updated_at`, unique `(trip_id, sequence)`).
   No type enum.
2. **Model — `Consignment`.** Add nullable `pickup_stop_id` / `delivery_stop_id` FKs → `trip_stops.id`,
   and nullable `load_priority` int. Keep existing `origin/destination_precinct_id`.
3. **Model — `Trip`.** Relax `client_organization_id`, `origin_precinct_id`,
   `destination_precinct_id` to `nullable=True`. Comment the *why* (per-consignment client; derived
   endpoints).
4. **Register.** Import `TripStop` in `db/models/__init__.py` from the `trips` import block.
5. **Schemas.** In `schemas/trips.py`: add `TripStopBase` / `TripStopCreate` / `TripStopRead`;
   add optional `stops: list[TripStopCreate] | None` to `TripCreateRequest`; relax
   `client_organization_id`/`origin_precinct_id`/`destination_precinct_id` to optional there; add
   `stops: list[TripStopRead]` to `TripDetailResponse`.
6. **Service.** In `trip_service.create_trip`: after the Trip row, build `TripStop` rows — from
   `payload.stops` if present, else synthesise two stops from origin/destination precincts and link a
   single consignment's pickup/delivery stops (A.3). Derive and set `Trip.origin/destination_precinct_id`
   from the earliest / latest stop. Add stops to the returned `TripDetailResponse`. Keep the existing
   flush ordering. Validate `pickup_stop.sequence < delivery_stop.sequence` for linked consignments.
7. **Migration.** Autogenerate, then hand-verify: creates `trip_stops` table, adds the three
   `consignments` columns (two FKs + `load_priority`), alters the three `trips` columns to nullable.
   Add a **data backfill** in `upgrade()`: for each existing trip, insert two `TripStop` rows from its
   current origin/destination. Confirm `downgrade()` reverses cleanly.
8. **Tests.** Unit: synthesise-stops produces correct sequence + consignment links; model nullable
   defaults; `pickup.sequence < delivery.sequence` validation. Integration: (a) create trip with
   explicit multi-stop + multi-consignment (incl. a stop that is origin for one consignment and
   destination for another) → stops persisted in order, roles resolve, endpoints derived; (b) single-leg
   create with no `stops` → two stops synthesised, behaves as before; (c) existing trip-detail read
   still returns 200 with `stops`.
